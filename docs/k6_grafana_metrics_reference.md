# PostForge 부하 테스트 분석 가이드

---

## Part 1. k6 용어 사전

### 핵심 개념

| 용어 | 의미 | 쉽게 말하면 |
|------|------|-------------|
| VU (Virtual User) | 가상 유저. 동시 접속자 수 | "30명이 동시에 사이트 쓰는 중" |
| iteration | VU 1명이 시나리오를 처음~끝 1바퀴 돈 것 | "1명이 목록보고 → 상세보고 → 검색하고 를 1번 완주" |
| scenario | 테스트 실행 방식 정의 (몇 명, 얼마 동안) | "30명을 4분 동안 돌려라" |
| stages | VU 수를 시간에 따라 조절하는 설정 | "0→10→30→30→0 순서로 사람 수 조절" |
| ramping-vus | stages에 따라 VU를 점진적으로 올리고 내리는 방식 | "한꺼번에 30명 넣지 않고 서서히 늘림" |
| group | 관련된 요청을 묶은 시나리오 단위 | "게시글 목록 조회"라는 이름표 붙인 것 |
| tags | 요청에 라벨을 붙여서 결과를 분리 | "이 요청은 GET /posts야" 라고 표시 |
| checks | 응답이 기대 조건을 만족하는지 확인 | "status 200 맞아?" 같은 검증 |
| thresholds | 미리 설정한 합격 기준 | "p95가 500ms 넘으면 불합격" |

### 응답 시간 지표

| 용어 | 의미 | 쉽게 말하면 |
|------|------|-------------|
| avg | 전체 요청의 평균 응답 시간 | 극단값에 왜곡됨. 단독으로 보면 위험 |
| med (median) | 전체 응답 중 정확히 가운데 값 | avg보다 실제 체감에 가까움 |
| p(90) | 100명 중 90명은 이 시간 안에 응답 받음 | "90%의 사용자 경험" |
| p(95) | 100명 중 95명은 이 시간 안에 응답 받음 | 가장 많이 쓰는 성능 기준 |
| p(99) | 100명 중 99명은 이 시간 안에 응답 받음 | 거의 최악 케이스 |
| max | 가장 느렸던 1건 | GC Pause, 커넥션 대기 등 일시적 문제 |
| min | 가장 빨랐던 1건 | 서버가 여유로울 때의 최적 성능 |

### HTTP 요청 분해 지표

k6는 요청 1건을 아래 단계로 쪼개서 측정함:

```
[blocked] → [connecting] → [tls_handshaking] → [sending] → [waiting] → [receiving]
|__________|______________|___________________|___________|___________|____________|
                                                                    = duration
```

| 용어 | 의미 | 느리면 의심할 것 |
|------|------|-----------------|
| http_req_blocked | 요청 보내기 전 대기 시간 | TCP/TLS 연결 대기. 커넥션 풀 부족 |
| http_req_connecting | TCP 연결 맺는 시간 | 네트워크 문제, 서버 소켓 부족 |
| http_req_tls_handshaking | HTTPS 암호화 연결 시간 | TLS 핸드셰이크 비용 (첫 연결만) |
| http_req_sending | 요청 데이터 → 서버 전송 시간 | 요청 body가 크거나 네트워크 느림 |
| http_req_waiting | 서버가 처리하는 시간 (TTFB) | DB 쿼리, 비즈니스 로직, GC 등 |
| http_req_receiving | 응답 데이터 수신 시간 | 응답 body가 크거나 네트워크 느림 |
| http_req_duration | 위 전체 합계 (핵심 지표) | 사용자가 체감하는 전체 시간 |

대부분의 경우 waiting이 duration의 99%를 차지함. waiting이 느리면 서버 쪽 문제.

### 기타 지표

| 용어 | 의미 |
|------|------|
| http_reqs | 총 요청 수 + 초당 처리량 (RPS) |
| http_req_failed | 실패율 (2xx가 아닌 응답) |
| iteration_duration | VU 1명이 1바퀴 도는 데 걸린 시간 (sleep 포함) |
| data_received / sent | 테스트 동안 주고받은 총 데이터량 |
| vus | 해당 시점의 활성 VU 수 |

## Part 2. k6 결과 정상 범위 기준표

### 응답 시간 기준 (일반적인 웹 API)

| 등급 | p95 기준 | 사용자 체감 |
|------|----------|------------|
| 우수 | 200ms 이하 | 즉각 반응, 쾌적 |
| 양호 | 200~500ms | 약간의 대기감 |
| 보통 | 500ms~1s | 체감되는 느림 |
| 나쁨 | 1~3s | 답답함, 이탈 시작 |
| 심각 | 3s 이상 | 사용 불가 수준 |

주의: 이건 일반 기준이고, 1GB/1vCPU 인스턴스에서 30VU면 "양호"가 현실적 목표임.

### 에러율 기준

| 등급 | http_req_failed | 의미 |
|------|-----------------|------|
| 정상 | 0~0.1% | 거의 에러 없음 |
| 양호 | 0.1~1% | 간헐적 에러 (허용 범위) |
| 주의 | 1~5% | 일부 요청 실패 |
| 심각 | 5% 이상 | 서버 과부하 또는 버그 |

### checks 통과율 기준

| 등급 | 통과율 | 의미 |
|------|--------|------|
| 정상 | 100% | 모든 응답이 기대대로 |
| 주의 | 95~99% | 간헐적으로 예상 외 응답 |
| 심각 | 95% 미만 | API 동작 자체에 문제 |

### RPS (초당 처리량)

절대적 기준은 없고, 부하 증가에 따른 변화 패턴이 중요:

| 패턴 | 의미 |
|------|------|
| VU↑ RPS↑ | 정상. 서버가 아직 여유 있음 |
| VU↑ RPS 정체 | 서버가 처리 한계에 도달 |
| VU↑ RPS↓ | 서버 과부하. 오히려 처리량이 줄어듦 |

### avg vs med 차이

| 패턴 | 의미 |
|------|------|
| avg ≒ med | 응답 시간이 고르게 분포 (정상) |
| avg > med × 1.5 | 소수의 느린 요청이 평균을 끌어올림 → max, p99 확인 |
| avg > med × 3 | 심각한 간헐적 지연 존재 → GC, 커넥션 풀, 락 경합 의심 |

### max 스파이크 판단

| 패턴 | 의미 |
|------|------|
| max ≤ p95 × 2 | 정상 범위의 변동 |
| max > p95 × 3 | 간헐적 문제 존재 (GC Pause, 커넥션 대기) |
| max > p95 × 10 | 심각한 일시적 장애 (GC Full, OOM 직전, 스레드 고갈) |

### http_req_blocked / connecting / tls_handshaking

| 지표 | 정상 | 주의 |
|------|------|------|
| blocked avg | 1ms 이하 | 10ms 이상이면 커넥션 풀 부족 의심 |
| connecting avg | 1ms 이하 | 지속적으로 높으면 네트워크 문제 |
| tls_handshaking avg | 10ms 이하 | 매 요청마다 높으면 keep-alive 미작동 |

## Part 3. Grafana 지표 정상 범위 기준표

### CPU

| 지표 | 정상 | 주의 | 위험 |
|------|------|------|------|
| CPU Usage | 70% 이하 | 70~90% | 90% 이상 (병목 확정) |
| Load | 코어 수 이하 | 코어 수의 1~2배 | 3배 이상 |

### GC

| 지표 | 정상 | 주의 | 위험 |
|------|------|------|------|
| GC Pressure | 1% 이하 | 1~5% | 5% 이상 (Heap 부족) |
| GC Pause max | 100ms 이하 | 100~500ms | 500ms 이상 |
| GC Collections | 부하에 비례 증가 | 급격히 증가 | 쉬지 않고 계속 실행 |

### 메모리

| 지표 | 정상 | 주의 | 위험 |
|------|------|------|------|
| Heap 사용률 | 60% 이하 | 60~80% | 80% 이상 |
| Heap 패턴 | 톱니바퀴 (오르내림) | 서서히 우상향 | GC 후에도 안 내려감 (누수) |
| Tenured Gen | 안정적, 평탄 | 서서히 증가 | 계속 증가 (누수) |
| Metaspace | 시작 후 안정화 | 런타임에 계속 증가 | 클래스 누수 |

### 스레드

| 지표 | 정상 | 주의 | 위험 |
|------|------|------|------|
| Thread States BLOCKED | 0 | 간헐적 발생 | 지속적 (락 경합) |
| Threads 수 | 부하에 비례, 안정적 | 부하 없는데 계속 증가 | 스레드 누수 |

## Part 4. 분석 체크리스트

### Step 1: k6 결과 전체 개요

- [ ] checks 100% 통과하는가?
  → 100% 미만이면 특정 API에서 예상 외 응답이 오고 있음
- [ ] 에러율 (expected_result:success) 1% 미만인가?
  → 예상된 실패(401, 토큰 없음)를 제외한 진짜 에러율
- [ ] RPS가 VU 증가에 비례해서 올라가는가?
  → 정체/감소하면 서버 처리 한계 도달

### Step 2: 엔드포인트별 응답 시간 분석

- [ ] p95 기준 가장 느린 엔드포인트 순위를 매겼는가?
  → p95가 사용자 95%의 체감 속도. avg는 왜곡될 수 있음
- [ ] avg와 med 차이가 1.5배 이상인 엔드포인트가 있는가?
  → 있으면 소수의 극단적으로 느린 요청이 존재한다는 뜻
- [ ] max가 p95의 3배 이상 튀는 엔드포인트가 있는가?
  → GC Pause, 커넥션 풀 대기, 락 경합 등 간헐적 문제 의심

### Step 3: Grafana I/O Overview

- [ ] Rate vs Duration 패턴 확인
  → Rate↑ Duration↑ = 처리 한계 / Rate↑ Duration 안정 = 여유 있음
- [ ] Errors 패널에서 5xx가 발생하는가?
  → 부하 중 5xx = 서버 과부하, 커넥션 풀 고갈, 예외
- [ ] Duration 스파이크 시점을 기록했는가?
  → 이 시점을 기준으로 CPU, GC, Heap과 대조하면 원인이 보임
- [ ] Utilisation (톰캣 스레드 사용률) 확인
  → 100%면 새 요청이 대기 상태

### Step 4: CPU 병목 확인

- [ ] CPU Usage가 90% 이상에 도달하는가?
  → 100% 도달 = CPU 병목 확정
- [ ] Load가 CPU 코어 수를 초과하는가?
  → 1vCPU인데 Load 5.7이면 5.7개 작업이 줄 서 있는 것
- [ ] CPU 100% 시점과 Duration 스파이크 시점이 겹치는가?
  → 겹치면 CPU가 응답 지연의 원인
- [ ] Thread States에서 BLOCKED가 지속적으로 발생하는가?
  → BLOCKED 많으면 락 경합 (synchronized, DB row lock)

### Step 5: GC 병목 확인

- [ ] GC Pause Duration이 k6 max 응답시간과 비슷한가?
  → GC Pause 80ms인데 max 5.6s면 GC는 원인 아님
  → GC Pause 3s인데 max 5.6s면 GC가 원인일 수 있음
- [ ] GC Pressure가 5%를 넘는가?
  → 5% 이상이면 Heap이 부족하다는 뜻
- [ ] GC 실행 횟수가 부하 중 급격히 증가하는가?
  → 객체 생성이 너무 많거나 Heap이 작음

### Step 6: 메모리 확인

- [ ] Heap이 톱니바퀴 패턴인가, 우상향인가?
  → 톱니바퀴 = 정상 / 우상향 = 메모리 누수
- [ ] 부하 중 Heap 사용률이 80%를 넘는가?
  → 넘으면 GC가 자주 발생하고 Pause도 길어짐
- [ ] Tenured Gen(Old Gen)이 안정적인가?
  → 계속 증가하면 장수 객체가 쌓이는 중 = 누수 의심

### Step 7: 종합 판단 및 조치

- [ ] 병목 원인이 CPU / GC / 메모리 / I/O 중 무엇인가?
  → CPU 100% = CPU 병목
  → GC Pause 높음 + Heap 부족 = GC 병목
  → Heap 우상향 = 메모리 병목
  → CPU 여유인데 느림 = I/O 병목 (DB, 커넥션 풀)

- [ ] 원인에 맞는 조치를 적용했는가?
  → CPU 병목: BCrypt rounds 확인, 부하 테스트에서 로그인 비중 분리
  → GC 병목: -Xms/-Xmx 고정, GC 알고리즘 변경 (1vCPU면 Serial GC)
  → 메모리 병목: 누수 원인 추적, 객체 생성 최적화
  → I/O 병목: slow query 확인, HikariCP pool 크기 조정

- [ ] 조치 후 동일 시나리오로 재테스트했는가?
  → before/after 비교 (같은 VU, 같은 duration으로)

## Part 5. 현재 PostForge 분석 결과 요약 (2026-03-10)

### 테스트 환경
- 서버: Oracle Cloud 1vCPU / 1GB RAM
- k6: 30 VU, 4분 (warm-up 30s → 증가 1m → steady 2m → cool-down 30s)

### k6 결과

| 엔드포인트 | med | p95 | max | 판정 |
|-----------|-----|-----|-----|------|
| POST /auth/login | 2.09s | 3.6s | 5.59s | 심각 |
| GET /posts | 502ms | 1.73s | 3.6s | 나쁨 |
| GET /posts?keyword | 382ms | 1.59s | 3.69s | 나쁨 |
| GET /posts/{id} | 384ms | 1.59s | 2.79s | 나쁨 |
| GET /comments | 209ms | 1.09s | 3.02s | 보통~나쁨 |
| POST /token/reissue | 196ms | 705ms | 1.3s | 보통 |

### Grafana 결과

| 지표 | 측정값 | 판정 |
|------|--------|------|
| CPU Usage | max 100% | 🔴 병목 |
| Load | max 5.7 | 🔴 코어의 5.7배 |
| GC Pressure | 0.1% | ✅ 정상 |
| GC Pause max | 80ms | ✅ 정상 |
| Heap 사용률 | 38% | ✅ 여유 |
| Heap 패턴 | 톱니바퀴 | ✅ 정상 |
| Tenured Gen | 안정적 | ✅ 누수 없음 |

### 결론

병목 원인: **CPU (100% 포화)**
근거: Load 5.7 (1코어 대비 5.7배 대기), GC/메모리는 정상
주요 원인: BCrypt 해싱이 1vCPU를 점유 → 나머지 읽기 요청까지 줄 서서 대기
