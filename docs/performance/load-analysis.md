# PostForge 부하 테스트 및 병목 분석 보고서

## 0. 테스트 환경 및 개요
- **서버 사양:** Oracle Cloud ARM 1vCPU / 1GB RAM
- **애플리케이션:** Spring Boot 3.x + PostgreSQL
- **부하 도구:** k6 (총 30 VU, 4분간 진행)
- **모니터링:** Prometheus + Grafana (Spring Boot Actuator, Micrometer)
  - 모니터링은 다른 클라우드에 구현

---

## 1. 문제 발생: 단일 시나리오 부하 테스트
초기 테스트는 30명의 가상 사용자(VU)가 **읽기 API와 로그인 API를 동일한 비중으로 호출**하도록 구성했습니다.

### 테스트 결과 및 현상
- **CPU 포화 상태:** Grafana 확인 결과 CPU Usage가 **100%** 에 도달했으며, Load Average는 **12.0** (1코어 기준 대기열 12개)까지 치솟았습니다.
- **처리량 정체:** 부하 초반에 상승하던 Rate(처리량)가 곧바로 한계에 부딪혀 정체 및 하락했습니다.
- **극심한 응답 지연:** 전체 API의 평균 응답 시간(AVG)은 170ms였으나, 최대 응답 시간(MAX)은 3.51s로 **약 20배의 편차**가 발생했습니다.

### 원인 분석 (CPU 병목)
원인은 **로그인 과정의 BCrypt 해싱 연산**이었습니다. (strength 10 설정 시 약 1,024번의 반복 해싱 수행)
1vCPU 환경에서 모든 VU가 매 반복마다 CPU-intensive한 해싱 작업을 수행하면서 CPU 독점 현상이 발생했고, 이로 인해 **비교적 가벼운 읽기 API 요청들까지 큐(Queue)에 쌓이며 연쇄적인 지연(med 1.98s)이 발생**한 것입니다.

## 2. 환경 조정: 실제 트래픽 비율을 반영한 시나리오 분리
원래 의도했던 애플리케이션의 성능을 정확히 측정하기 위해, 부하 시나리오를 **실제 서비스 트래픽 비율에 가깝게 조정(읽기 25 VU + 로그인 5 VU)** 하여 재테스트를 진행했습니다.

### 시나리오 분리 후 변화 (Before ➔ After)

| 모니터링 지표 | 단일 시나리오 (Before) | 시나리오 분리 (After) | 개선 효과                 |
| :--- | :--- | :--- |:----------------------|
| **CPU Usage (Max)** | 100% | **59%** | **여유 확보 (41%p 감소)** |
| **System Load (Max)** | 12.0 | **0.7** | **대기열 해소 (17배 감소)**   |
| **RPS (처리량)** | 14.66 | **23.75** | **1.6배 증가**           |
| **Max Duration** | 3.51s | **182ms** | **19배 개선**            |

### 주요 API 응답 속도 개선 (p95 기준)
| API 엔드포인트 | 단일 시나리오 (Before) | 시나리오 분리 (After) | 개선 수치         |
| :--- | :--- | :--- |:--------------|
| `GET /posts/{id}` | 1.60s | **205ms** | **7.8배 개선**   |
| `GET /posts/{id}/comments` | 1.28s | **189ms** | **6.8배 개선**   |
| `GET /posts` | 1.89s | **508ms** | **3.7배 개선** |

로그인 요청 비율을 조정하자 CPU 병목이 해소되었으며, 밀려있던 읽기 API들의 응답 속도가 최대 7.8배까지 대폭 개선되었습니다.

## 3. 결론 및 Next Step (새로운 병목의 발견)

### 결론 (인사이트)
비현실적인 단일 부하 시나리오가 유발한 **CPU 100% 포화 현상(가짜 병목)** 을 걷어냄으로써, 애플리케이션 전체의 처리량(RPS)을 1.6배 향상시키고 시스템 지표를 안정화(CPU 59%)할 수 있었습니다.

### 추가 발견 (Hidden Bottleneck)
시스템이 안정화된 이후 데이터를 세부적으로 분석한 결과, **새로운 병목 지점**을 발견했습니다. 
다른 단건 조회 API(`GET /posts/{id}`)의 중앙값(med)이 **16~31ms**인 것에 비해, 다건 조회인 `GET /posts` API만 유독 **90ms** 이상으로 지연되고 있었습니다. 

디버그 로그(`org.hibernate.SQL: DEBUG`)를 통해 분석한 결과, `GET /posts` 요청 1건당 **11번의 추가 쿼리가 발생하는 N+1 문제**를 확인했습니다. CPU 자원이 넉넉함에도 특정 API만 느리다면, 이는 전형적인 I/O(DB 쿼리) 병목 현상입니다.

### 향후 개선 계획 (Next Step)
1. **쿼리 최적화:** Spring Data JPA의 `Fetch Join` 또는 `@EntityGraph`를 적용하여 `GET /posts` 조회 시 발생하는 N+1 다중 쿼리를 단일 쿼리로 최적화합니다.
2. **성능 재검증:** 쿼리 튜닝 후 동일한 부하 테스트(시나리오 분리 버전)를 다시 수행하여 `GET /posts` API의 응답 속도(med, p95) 개선 폭을 검증할 예정입니다.
