# Spring Backend 피드백 가이드

## 문서 목적

이 문서는 PostForge 백엔드에서 나온 리뷰 피드백을 일회성 메모로 끝내지 않고,
앞으로 구현/리팩터링/리뷰 때 반복해서 적용할 수 있는 기준으로 정리한 문서다.

핵심 목표는 아래와 같다.

- 스프링 기본기를 우선해서 설계한다.
- 보안 책임을 한 층에 명확하게 둔다.
- 운영 기본값을 안전하게 유지한다.
- 상태성 로직은 실패 시 재처리를 기본으로 설계한다.
- 중복 규칙은 초기에 공통화한다.

---

## 1. 이 프로젝트에서 우선 적용할 원칙

### 1-1. 인증과 인가는 한 층에 책임을 모은다

`HttpSecurity`와 `@PreAuthorize`를 둘 다 쓰는 것은 문제가 아니다.
문제는 같은 책임을 양쪽에 중복으로 나눠서 두는 것이다.

권장 기준:

- `HttpSecurity`: URL 단위의 거친 접근 제어
- `@PreAuthorize`: 도메인/리소스 단위의 세밀한 인가

예시:

- `GET /posts/**`는 공개인지 인증 필요인지: `HttpSecurity`
- "이 게시글을 수정할 수 있는가": `@PreAuthorize`

피해야 할 패턴:

- `HttpSecurity`에서 `permitAll`
- 컨트롤러에서 같은 엔드포인트에 다시 `hasRole('USER')`

이렇게 되면 실제 정책과 코드가 어긋나서 나중에 추론이 매우 어려워진다.

### 1-2. 운영 기본값은 안전하게, 개발 편의는 프로필로 분리한다

공유 `application.yml`에는 안전한 기본값만 둔다.

권장 기준:

- 운영에 위험한 값은 base 설정에 두지 않는다.
- 개발 편의용 설정은 `application-local.yml`로 분리한다.
- 비밀값은 env var 또는 시크릿 매니저로 관리한다.

base 설정에 두면 안 되는 대표 예:

- `ddl-auto: update`
- `show-sql: true`
- `DEBUG`, `TRACE` 로그
- 평문 자격증명
- 강제 `spring.profiles.active`

### 1-3. Redis 상태성 로직은 "실패 시 재처리"를 먼저 설계한다

Redis를 캐시처럼만 쓰면 단순하지만,
조회수 버퍼링/임시 토큰/작업 큐처럼 상태를 들고 시작하면 설계 난이도가 급격히 올라간다.

반드시 먼저 확인할 질문:

- 중간 실패 시 데이터가 유실되는가
- 같은 작업이 두 번 처리돼도 안전한가
- 처리 중 상태와 완료 상태가 분리되는가
- 실패 항목을 다시 주울 수 있는가

권장 기준:

- 처리 전 삭제하지 않는다.
- 성공 후 제거한다.
- 중복 실행에 안전하게 만든다.
- 실패 항목은 재시도 가능하게 남겨둔다.

### 1-4. 같은 규칙이 2곳 이상 나오면 빨리 공통화한다

`post like`와 `comment like`처럼
"입력 검증 → 중복 확인 → 저장/삭제 → count 집계 → 응답 생성" 흐름이 거의 같다면,
처음부터 공통화 후보로 본다.

권장 기준:

- 흐름이 같고 대상만 다르면 템플릿 메서드
- 흐름 일부만 같으면 조합용 컴포넌트
- 공통 규칙이 늘어날수록 중복 구현은 빠르게 제거

### 1-5. S3 presign은 업로드 전 검증과 업로드 후 검증을 분리한다

Presigned URL은 서버가 파일 바이트를 직접 받지 않는다.
즉 로컬 업로드처럼 업로드 전에 완전 검증을 할 수 없다.

따라서 보안을 2단계로 나눠야 한다.

업로드 전:

- 확장자 allowlist
- declared content-type allowlist
- key prefix 제한
- 짧은 만료 시간
- 업로드 최대 크기 제한

업로드 후:

- 실제 MIME 검증
- 필요 시 악성코드 검사
- 검증 완료 전 공개 차단
- 검증 통과 후만 게시글 연결

## 2. 스프링 기본기 체크리스트

아래는 이 프로젝트에서 "스프링 기본기를 잘 쓰고 있는가"를 볼 때 기준이 되는 항목이다.

### 2-1. DI / 빈 설계

- 생성자 주입을 기본으로 사용한다.
- 필드 주입보다 생성자 주입을 선호한다.
- `new`로 직접 만들기보다 빈으로 관리할 수 있는 것은 빈으로 둔다.
- 설정 객체는 `@ConfigurationProperties`를 우선 검토한다.

### 2-2. 웹 계층

- `@RestController`는 HTTP 입출력 책임만 가진다.
- 요청 검증은 `@Valid`와 DTO에서 처리한다.
- 비즈니스 규칙은 서비스로 넘긴다.
- 응답 상태코드가 도메인 의미와 맞는지 확인한다.

### 2-3. 예외 처리

- 예외 응답은 `@RestControllerAdvice`로 통합한다.
- 서비스에서 HTTP 응답 객체를 직접 만지는 일을 최소화한다.
- 커스텀 예외는 도메인 의미가 분명해야 한다.

### 2-4. 보안

- URL 접근 정책은 `SecurityFilterChain`에서 정리한다.
- 소유권/도메인 인가는 `@PreAuthorize`에서 처리한다.
- 비밀번호 비교는 반드시 `PasswordEncoder`를 사용한다.
- Principal 모델은 한 가지 방식으로 일관성 있게 가져간다.

### 2-5. 설정

- env 값은 `@Value`보다 `@ConfigurationProperties`를 우선 검토한다.
- 설정 누락을 `@Validated`로 앱 시작 시점에 막는다.
- 프로필별 차이는 파일 또는 env override로 분리한다.

### 2-6. 트랜잭션

- 쓰기 메서드에만 `@Transactional`을 두고, 읽기는 `readOnly = true`를 검토한다.
- 트랜잭션 경계는 서비스에서 잡는다.
- 컨트롤러에서 트랜잭션을 관리하지 않는다.

### 2-7. 테스트

- 컨트롤러는 `@WebMvcTest`
- 서비스는 단위 테스트
- 실제 흐름은 `@SpringBootTest` 또는 통합 테스트
- 보안/설정/예외 흐름도 테스트 대상에 포함한다

## 3. JPA 기본기 체크리스트

JPA는 "쿼리가 돌아간다"보다
"엔티티 경계와 조회 전략이 맞는가"가 더 중요하다.

### 3-1. 엔티티 설계

- 엔티티는 도메인 상태와 행위를 함께 가진다.
- 무분별한 setter를 열지 않는다.
- 생성/수정 메서드의 의도가 명확하다.

### 3-2. 연관관계

- 기본은 `LAZY`
- 양방향은 꼭 필요한 경우만
- 연관관계 편의 메서드는 의도를 분명히 한다

### 3-3. 조회 성능

- N+1을 항상 의심한다.
- 목록 조회와 상세 조회는 같은 방식으로 보지 않는다.
- 필요 시 fetch join, projection, `@EntityGraph`를 사용한다.

### 3-4. 트랜잭션과 Dirty Checking

- 수정은 트랜잭션 안에서 엔티티 변경으로 처리한다.
- 불필요한 `save()` 반복 호출을 줄인다.
- 읽기 전용 메서드는 쓰기 의도가 없어야 한다.

### 3-5. 벌크 연산

- `update ... where ...` 같은 벌크 쿼리는 영속성 컨텍스트와 어긋날 수 있음을 인지한다.
- 벌크 연산 직후 같은 엔티티를 다시 읽을 때는 일관성을 점검한다.

### 3-6. 공통 필드

- `createdAt`, `updatedAt`, `createdBy` 같은 값은 Auditing으로 통일한다.
- 엔티티마다 같은 필드를 반복하지 않는다.

## 4. AOP를 꼭 써야 하는가

결론부터 말하면, 아니다.

AOP는 스프링 심화 기술이지만,
좋은 백엔드는 AOP를 많이 쓴 코드보다
기본 책임 분리가 잘 된 코드일 가능성이 더 높다.

### AOP가 적합한 경우

- 공통 로깅
- 실행 시간 측정
- 감사 로그
- 반복되는 공통 정책
- 여러 메서드에 걸쳐 공통으로 넣어야 하는 부가기능

### AOP보다 기본 기능이 먼저인 경우

- 인증/인가: Spring Security
- 요청 검증: Bean Validation
- 예외 응답: `@RestControllerAdvice`
- 트랜잭션: `@Transactional`
- 요청 전처리: Filter / Interceptor
- 설정 관리: `@ConfigurationProperties`

기준:

- 비즈니스 로직이면 AOP보다 서비스/도메인으로
- 횡단 관심사이면 AOP 검토

즉 "AOP를 안 써서 부족한가?"보다
"기본 기능으로 충분한데 AOP로 복잡하게 만들고 있지 않은가?"를 먼저 본다.

---

## 5. 이 프로젝트에서 AOP 없이도 충분한 영역

현재 PostForge 기준으로는 아래 영역은 AOP 없이도 충분히 관리 가능하다.

- 인증/인가
- 예외 응답 통합
- DTO 검증
- 트랜잭션 경계
- 파일 업로드 정책
- 조회수/좋아요 서비스 흐름

굳이 AOP를 검토할 수 있는 영역:

- API 실행 시간 로깅
- 관리자/운영용 감사 로그
- 특정 패키지 공통 메트릭 수집

---

## 6. 이 프로젝트에서 우선 강하게 가져가야 할 스프링/JPA 기본기

우선순위는 아래 순서다.

1. Security 책임 분리
2. `@ConfigurationProperties` 기반 설정 정리
3. 트랜잭션 경계 명확화
4. JPA 조회 전략과 N+1 감지
5. Redis 상태성 로직의 재처리 설계
6. 공통 규칙 추상화

즉 심화 기술을 더 넣기 전에,
아래 질문에 모두 "예"라고 답할 수 있어야 한다.

- 이 엔드포인트의 공개/인증 정책을 한눈에 설명할 수 있는가
- 운영 기본 설정이 안전한가
- 실패 시 다시 처리해도 깨지지 않는가
- 같은 규칙이 여러 곳에서 복사되지 않았는가
- JPA 쿼리와 연관관계를 설명할 수 있는가

---

## 7. 코드 리뷰 때 바로 쓰는 질문

리뷰할 때 아래 질문을 체크리스트처럼 사용한다.

### 보안

- 이 엔드포인트 정책이 `HttpSecurity`와 `@PreAuthorize`에서 충돌하지 않는가
- 비밀값/운영 계정이 안전하게 관리되는가
- presigned 업로드 전후 검증이 분리되어 있는가

### 스프링 기본기

- 컨트롤러가 서비스 책임까지 먹고 있지 않은가
- 설정이 `@Value` 난사 대신 구조적으로 묶여 있는가
- 예외 처리가 일관적인가

### JPA

- 목록/상세 조회에서 N+1 위험이 없는가
- 벌크 업데이트 뒤에 영속성 컨텍스트 일관성이 괜찮은가
- 엔티티 연관관계가 필요한 만큼만 열려 있는가

### 상태성 로직

- Redis/스케줄러/비동기 처리에서 실패 항목이 유실되지 않는가
- 재시도해도 안전한가

### 구조

- post/comment처럼 거의 같은 흐름이 중복되어 있지 않은가
- 공통 규칙을 한 곳에서 바꿀 수 있는가

---

## 8. 한 줄 결론

좋은 스프링 프로젝트는
"심화 기술을 많이 쓴 프로젝트"가 아니라
"기본 책임이 올바른 층에 있고, 운영 기본값이 안전하며, 실패 시 복구 가능하고, 중복 규칙이 잘 정리된 프로젝트"다.
